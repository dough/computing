The OPERATIONS language **ACAC**

Douglas Harris

Now that a network has been constructed we need to think about how it will operate. Even more important (and not always done) is to think about how it SHOULD be operated before it is built.

The phrase “be operated” is vital; even an automated system is operated, if properly done that is exactly what we will be discussing with the language discussed here.

The language is called ACAC (or CACA), and the letters remind us of the basic tasks of Creating and Absorbing information.

When in PEPE we talk about push and pull very obvious questions arise. There is information and there are components, and the terms relate to both. Of course you may know the component the information is coming from or going to, but what about the information itself?

Two new concepts enter our thought: a Source is something information comes from, and a Sink something it goes to. Both sound perhaps like a storage unit, in PEPE terms a Pipe, and that is a proper passive view.

A Source has been pushed already, so we can just pull it when we need to, perhaps many times and from man places, perhaps indirectly through Pulls. and that is the passive view. But we will always get the same information. It may change because the source is replaced, but we have no control.

A passive Sink is similar in reverse. We push a Sink whenever we are supposed to, and the information is potentially gone forever. If retrieval were ever to be desired a Pipe would have been used in the design.

My personal model for Source and Sink has always come from the model used in network benchmarking measurements for speed. You want to consider only communications time, not processing, so a Sink just throws the data away, never even writing it into memory or storage, and a Source just gives out predetermined data that may even be constant, so the benchmark will be repeatably the same.

When it comes to active Sources and Sinks, a whole new question emerges. Do we need to tell them we want action, or do we just go ahead. For example on output we may want an acknowledgment that a buffer has been pulled before we push new data, and on input we may want to know that new data has been pushed dand should now be pulled.` Environmental measurements are often a good example: you may only need to show the current temperature, not to remark “its getting hotter!” or to ask “did you see how hot it was?”.

These considerations say that “Have You Created?” may be important to ask before Accept, and “Have You Accepted?” before Create.

A similar set of issues arises with Push and Pull:

  • When a pull operation is done is it to be expected that the right information was already in place,

  • or is it expected that the Pull will notify upstream that new data should be pushed.

  • Does a push just assume the destination is prepared

  • or does it politely notify downstream that the old data should be pulled?
